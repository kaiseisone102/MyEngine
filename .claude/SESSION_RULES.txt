================ MyEngine セッション最重要ルール (hook 強制注入) ================
このテキストは SessionStart で毎セッション自動注入される。設計でも実装でも従う。

【設計するとき (実装前・調査/提案も含む)】
- まず開発資料のコーディングルールを読んでから設計する (START_HERE §0 全項 / §1.5 設計方針 / Work_Protocol §1-2a)。
- §0-8: 資料に答えがある判断は二択を出さず、資料の答えを実行する。
- §0-9: 設計提示前に ①最新技術である根拠 ②既存に引っ張られて妥協していない根拠 を文面に明記。
- GPU 性能・現状の規模に関わらず最新技術を実装する (START_HERE L40/L44/L60)。規模を理由に躊躇しない。
- 設計時にも「実装するときの留意点」(下記 §3 系) を先に読む。

【実装するとき】
- §0-10 推測でコードを書くな: 関数シグネチャ/メンバ/struct/throw/include を実ソース Read してから書く。読んだふり禁止。
- §3-1a: types.h など struct サイズが変わる変更をしたら必ず clean rebuild (--target clean → 再ビルド)。
- §3-1b: ソース変更後は exe/.spv のタイムスタンプで反映を確認 (error 0 だけで信用しない)。
- §3-2: 起動後 validation/VUID/leaked を全行確認 (先頭で切らない)。描画変更は user 目視依頼。
- build/commit は 1 メッセージ 1 ツール。commit は走らせて git rev-parse で hash 確認してから報告。
- 怖い変更 (types.h / shader 全置換 / pipeline layout / pass_chain 配線) は変更前後を提示して承認後に Edit。

【ツール発行と完了報告の作法 (2026-05-30 事故で確立・最優先で守る)】
- ツールを呼ぶときは「本文に説明を書いてからツール」をやめる。説明文ゼロで、1 メッセージ 1 ツールだけ発行する。
  理由: 本文に court / <invoke ...> のようなツール呼び出し「文字」を書くと、実際のツールとして発行されず空振りし、
  何も実行されないままユーザーにターンが戻る事故が多発した。複数ツールを並べるのも崩れる原因。1 個ずつ確実に。
- 「完了」「直した」「ビルド成功」「起動 OK」と書く前に、必ず証拠を 1 つ出す:
  編集 = grep で適用確認 / ビルド = error 0 + exe・.spv のタイムスタンプ更新 / 起動 = validation ログの実行行 /
  commit = git rev-parse で hash 確認。証拠を出せないものを完了と書かない。不確かなら「未確認」と書く。
- 失敗の原因を環境・フック・ユーザーのせいにしない。まず「自分が確認を飛ばした」を疑う (実際に何度もそれが原因だった)。

【編集は「時間差」で成立する — これを知らないと未適用を完了と誤報告する (観測ベース・未検証注意)】
- ※以下は 2026-05-30 セッションで「観測された挙動」であり、フック仕様を実ソースで裏付けたものではない
  (当時ディスクの .claude/hooks/*.ps1 には marker 処理が無く、実体フックの所在は未特定)。鵜呑みにせず、
  実際に弾かれたメッセージに書かれた手順をその場で読んで従うこと。
- 観測: ソース編集 (Edit/Write/MultiEdit) が PreToolUse で弾かれることがある:
  「[BLOCKED ...] .claude/.docs_read_marker missing -> edit DENIED」。
- 観測した解除手順: 開発資料を読み、該当ルールを引用してから marker を PowerShell で作成
  (New-Item -ItemType File -Force '<repo>\.claude\.docs_read_marker')。その後 Edit を再実行すると通った。
  = 「最初の Edit はブロック → marker 作成 → 再 Edit で適用」という時間差で編集が成立した。
- 観測: marker はターンをまたぐと再び無効化され、再度ブロックされた (クリアの正確な契機は未検証)。
  ブロックされたら、その場のメッセージの指示どおり作り直す。
- 重要 (これは確実): ブロックされた Edit は適用されていない。必ず grep で内容を確認してから「適用済み」と扱う。
  ブロックに気づかず未適用のまま「完了」と報告し、中身の無い commit を作る事故を実際に起こした (二度とやらない)。
